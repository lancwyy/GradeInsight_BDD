# User Story (缺陷示例): 資料備份與復原

## 基本資訊
- **ID**: US-010 **[缺陷示例]**
- **版本**: Release_3
- **狀態**: Planned
- **優先順序**: Medium
- **缺陷類型**: 需求不完整/太模糊

## User Story 描述

**作為** 授課教師

**我想要** 系統能進行備份

**以便於** 防止資料丟失

## 接受標準

1. [ ] 支援備份功能
2. [ ] 支援復原功能
3. [ ] 備份應該按某種方式進行
4. [ ] 復原應該快速

---

## 缺陷分析

此User Story存在以下嚴重的不完整性和模糊性問題：

1. **關鍵資訊缺失**：
   - 什麼資料需要備份？（全部成績、部分成績、設定資訊？）
   - 備份頻率應該是多少？（每天、每周、手動觸發？）
   - 備份保留多少套副本？（最近10次、30天內？）
   - 備份儲存在哪裡？（本地、雲端、多個位置？）

2. **復原需求不清楚**：
   - 復原到哪個時間點？
   - 是復原部分資料還是全部資料？
   - 如果復原後發現不對應該如何回滾？
   - 復原時是否要保留最新的資料？

3. **接受標準無法驗證**：
   - 「按某種方式進行」太模糊，無法實現或測試
   - 「快速復原」沒有定義時間標準（1秒？1分鐘？）
   - 沒有失敗情況和邊界條件的描述

4. **潛在的非功能性需求未指定**：
   - 備份檔案的安全性要求（加密？）
   - 備份可用性保證（復原成功率？）
   - 性能要求（備份不應該影響系統正常使用？）
   - 法規要求（資料保留期？稽核紀錄？）

5. **使用場景不清楚**：
   - 何時觸發備份？（自動還是手動？）
   - 教授端還是系統管理員操作？
   - 緊急情況下復原的優先順序？

### 改進建議

應該分解為更具體的User Stories：

1. **自動備份策略Story**
   - 每天午夜自動備份系統所有成績資料
   - 保留最近30天的備份副本
   - 備份檔案儲存在加密的伺服器上

2. **手動備份Story**
   - 教授可按需匯出成績資料為加密備份檔案
   - 備份檔案包含時間戳和版本資訊

3. **資料復原Story**
   - 系統管理員可透過選擇備份點進行復原
   - 復原應在5分鐘內完成
   - 復原前應進行驗證和確認

4. **復原驗證Story**
   - 復原後應驗證資料完整性
   - 提供復原變更摘要報告
